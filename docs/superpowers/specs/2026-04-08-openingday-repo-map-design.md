# OpeningDay Repo Map & Auto-Setup — Design Spec

**Date:** 2026-04-08
**Status:** Draft

## Overview

Add a persistent, versioned structural snapshot of the entire codebase (repo map) that lives alongside the existing code tree. The repo map improves worker effectiveness by providing landscape awareness, strengthens seeder planning for brownfield projects, and auto-detects project environment to eliminate manual setup steps.

## Core Principle

All AI-to-AI content uses wire mode: compressed, factual, no prose. Field names abbreviated. Descriptions terse. If only machines read it, no reason to be fancy.

## Repo Map vs Code Tree

Two files, clean separation:

|             | `repo-map.json`                        | `code-tree.json`                      |
| ----------- | -------------------------------------- | ------------------------------------- |
| **What**    | Every file in the repo                 | Files being created/modified by tasks |
| **Source**  | Scanned from real code                 | Generated by seeder for planned work  |
| **Updates** | Incremental after each task completes  | Seeder + worker results               |
| **Git**     | Committed (version-controlled history) | Gitignored (ephemeral task state)     |
| **Purpose** | Landscape awareness, smarter planning  | Worker task dispatch                  |

## Scan Tiers

Three levels, user chooses based on project complexity and budget:

| Tier         | Flag           | Captures                                                                 | Cost         | Use case                        |
| ------------ | -------------- | ------------------------------------------------------------------------ | ------------ | ------------------------------- |
| **lite**     | `--depth lite` | File paths, exports, imports, module boundaries                          | ~2s, $0      | Quick tasks, small features     |
| **standard** | default        | Lite + package.json, tsconfig, .env vars, test patterns, infra, monorepo | ~5s, $0      | Most projects                   |
| **deep**     | `--depth deep` | Standard + AI-generated wire-mode summaries per module                   | ~30s, ~$0.50 | Complex repos, first onboarding |

## Data Model

Wire-mode compressed format. Only AI reads this.

```json
{
  "v": 1,
  "scannedAt": "2026-04-08T10:00:00Z",
  "depth": "standard",
  "env": {
    "pm": "pnpm",
    "test": "vitest",
    "lint": "eslint",
    "ts": true,
    "monorepo": true,
    "workspaces": ["packages/*"],
    "infra": "sst"
  },
  "deps": ["hono", "electrodb", "aws-lambda-powertools", "sst"],
  "modules": [
    {
      "p": "packages/core",
      "d": "shared biz logic: ElectroDB entities, auth, shopify client",
      "fc": 34,
      "k": ["ElectroDB", "auth", "shopify"],
      "files": [
        {
          "p": "packages/core/src/refunds/entity.ts",
          "ex": [{ "n": "RefundEntity", "s": "Entity<...>" }],
          "im": [{ "f": "core/dynamodb/config", "n": ["table"] }],
          "loc": 142
        }
      ]
    }
  ],
  "infra": {
    "resources": ["DynamoDB:MainTable", "S3:RefundExports", "SNS:NewRefundTopic"]
  }
}
```

Field key: `p`=path, `d`=description, `fc`=file count, `k`=keywords, `ex`=exports, `im`=imports, `s`=signature, `n`=name(s), `f`=from, `loc`=lines of code.

Deep tier adds `d` fields on modules via one AI pass. Wire-mode summaries only:

```json
{
  "p": "packages/core/src/refunds",
  "d": "refund lifecycle: batch upload→review→approve→issue gift card/credit→klaviyo email"
}
```

## Integration

### Context Builder

Updated signature:

```
buildContext(workTree, codeTree, repoMap, config, taskId, memory, rules)
```

Worker context package gains two new fields:

- **`landscape`** — Compressed full repo index (~200 tokens). Module names, file counts, key exports. Workers know the repo shape.
- **`relevant`** — Detailed repo map entries for files sharing modules/imports with the task (~50-200 tokens). Workers see nearby utilities.

This prevents: reinventing existing utilities, creating inconsistent patterns, putting files in wrong locations.

### Seeder

Updated signature:

```
seedFromSpec(specText, projectName, cwd, budgetUsd, repoMap?)
```

When repo map provided, seeder prompt includes compressed map. AI generates tasks that reference real files, reuse existing utilities, follow existing directory conventions.

### Orchestrator — Incremental Refresh

After each task passes gates and merges:

1. Get `output.filesChanged` from worker
2. Re-scan only changed files (regex, ~milliseconds)
3. Update repo map entries for those files
4. Persist updated map
5. Next task sees fresh data

### Wire Mode

`toWirePrompt()` updated to include `landscape` and `relevant` from enriched context. Compressed field names as always.

### Gates — Environment Aware

Gates read `env` from repo map:

- L1 automated: runs `{env.pm} test` not hardcoded `npx vitest run`
- L1 lint: runs `{env.pm} lint` not hardcoded `npx eslint`
- Workers use `{env.pm} add` not `npm install`

## Auto-Setup

On `openingday new` or `openingday init`, zero manual steps:

| Step                               | Detection method                                                           | Stored in        |
| ---------------------------------- | -------------------------------------------------------------------------- | ---------------- |
| Add `.openingday/` to `.gitignore` | Append if not present                                                      | `.gitignore`     |
| Track `repo-map.json` in git       | Add `!.openingday/repo-map.json` exception                                 | `.gitignore`     |
| Package manager                    | Lockfile detection: pnpm-lock.yaml, yarn.lock, bun.lock, package-lock.json | `env.pm`         |
| Test runner                        | devDeps + config files: vitest.config, jest.config, .mocharc               | `env.test`       |
| Linter                             | devDeps + config: eslint.config, biome.json                                | `env.lint`       |
| TypeScript                         | `tsconfig.json` exists                                                     | `env.ts`         |
| Monorepo                           | pnpm-workspace.yaml, workspaces in package.json, turbo.json, lerna.json    | `env.monorepo`   |
| Workspace packages                 | Parse workspace globs, list actual dirs                                    | `env.workspaces` |
| Dependencies                       | Read package.json deps (root + workspace packages)                         | `deps`           |
| Infrastructure                     | sst.config.ts, serverless.yml, cdk.json, terraform/, Dockerfile            | `env.infra`      |

## CLI Commands

| Command                                          | Change                                              |
| ------------------------------------------------ | --------------------------------------------------- |
| `openingday new`                                 | Modified — scan at chosen tier after creation       |
| `openingday init`                                | Modified — scan at default tier, auto-setup         |
| `openingday scan [--depth lite\|standard\|deep]` | **New** — manual repo map refresh                   |
| `openingday run`                                 | Modified — repo map in context, incremental refresh |

### `openingday scan`

```bash
openingday scan                # standard (default)
openingday scan --depth lite   # structure only
openingday scan --depth deep   # includes AI summaries
```

## Version Control

`repo-map.json` is the only `.openingday/` file committed to git.

Flow:

1. `openingday new` → scan → write `repo-map.json`
2. `git add .openingday/repo-map.json && git commit` → structural snapshot
3. `openingday run` → workers modify code → incremental map refresh
4. After run: `git diff .openingday/repo-map.json` shows structural changes
5. Commit updated map with feature branch

Long-term: `git log -p .openingday/repo-map.json` = architectural evolution history.

## Multi-Developer Usage

OpeningDay is designed for **solo dev per branch**. Same model as Claude Code — one agent session per working copy.

**Works:**

- Dev A on `feat/billing`, Dev B on `feat/search` — different branches, zero conflict
- Dev A finishes, merges. Dev B rebases, scans, continues — clean
- Multiple devs using OpeningDay on same repo, different features — fine

**Would conflict:**

- Two devs running `openingday run` targeting same files simultaneously — git merge conflicts

**Recommendation:** Feature branches. Coordinate via git, not OpeningDay. Multi-dev coordination layer deferred to Phase 4+.
